What you can do with Bake-off

Put three models on one task, under the same conditions, and get a recommendation. Everything below is reachable from the omnibox, the + menu, or the four Settings tabs.

A speed test, not a benchmark
  • Think speedtest.netA bake-off is a point-in-time reading of YOUR models on YOUR hardware with YOUR task. Nobody asks a speed test to generalize - you just run it again when something changes. Results are meant to be current, not eternal.
  • Why not "benchmark"A benchmark invites the fair-comparison argument: workload, concurrency, quantization, serving stack, hardware, tail latency, and cost per watt all differ between labs, so any general ranking can be contested. A bake-off answers that by construction - same task, same prompts, same rig, same grading, and exactly one thing varies: the model. The same-conditions strip on every report states it plainly.
  • What it is notNot an evaluation platform, not a model tuning studio, not a leaderboard. Bake-off never tunes, quantizes, or provisions anything - it measures what you already serve, as deployed, and recommends. Your platform stands things up; Bake-off tells you which one earned the slot.
  • Ephemeral by designGPUs get borrowed, models get swapped, engines get upgraded. The live board describes this lab today; the downloadable report is the record. When the answer matters again, run it again.
The flow
  • 1 · Power onThe wizard warms your lineup and shows each model going green. Fix anything red from the Go step.
  • 2 · Pick a taskQ&A sets test text models; Document sets send page images to vision models. The task picks the right kind of lineup for you.
  • 3 · Press PlayThe Set runs continuously - live tokens, speed, and accuracy per model.
  • 4 · Read the reportA recommendation with confidence, best-at-each awards, cost avoided, and a downloadable HTML report. The same-conditions strip says exactly what was held equal.

Tasks: pick one or bring your own
  • PickTap a card to preview and load it. Pin favorites to the top strip from the dots menu; recently played sits beside them. Show all reveals the rest of a section.
  • PasteType or paste questions straight into the omnibox - one per line, or Q:/A: pairs. Multi-line text becomes a task automatically.
  • LinkPaste a GitHub or GitLab file link (use the Raw view; .csv, .txt, .json). Project pages and direct data files work too. Public links only for now.
  • UploadThe + menu takes a .csv, .txt or .json from your computer.
  • Strict rulesColumns named __expected grade by exact rules (equals:, regex:) and several of them must ALL appear - key-point grading. Full formats: docs/task-format.md in the repo.
  • SaveAfter review, save as a reusable card. Deleted cards wait in Settings > Tasks until emptied.
Models & Hugging Face
  • Seat a modelTap a slot, then tap a model. The dots send it to a specific slot instead. A seated model wears its slot's color.
  • Add from HFPaste a huggingface.co model link or org/name id into the omnibox - it joins the catalog and prefetches to a fitting machine in the background.
  • TrendingRefresh trending models (+ menu) pulls today's text AND vision models. One-tap offers are filtered to what vLLM can boot and your card can hold - gated or oversized models are never offered.
  • Vision vs textA set compares one kind: with a vision set loaded, text models fold into a set-aside drawer (and the other way round). Tapping one switches the whole set.
Machines
  • Enroll over sshThe one-click path. Type the box's ssh address (like joe@192.168.128.200) into the search bar or Add a machine, press Enroll, and this dashboard sets the whole box up for you. Works even when the box's network cannot reach this dashboard - a private connection is created automatically.
  • Add by addressAny OpenAI-compatible /v1 endpoint - vLLM, TensorRT-LLM, SGLang, NIM, llama.cpp, LM Studio. A cluster is still one URL. Hosted NIMs take their nvapi- key here; it is stored with the machine.
  • BootloaderFor a box you can copy a file to but not ssh into from here: Generate bootloader (+ menu), copy the file over, run it once. The box installs vLLM, serves a fitting model, and enrolls itself.
  • UnenrollThe reverse, from the machine's menu: the box stops serving, everything Bake-off put on it is removed, its connection is taken down, and it leaves this list. Only downloaded model weights stay behind, so enrolling the box again later is quick.
  • With an agentAgent-equipped boxes report real hardware (the GPU chips on rows), can swap models on tap, and show what is cached on disk.
  • Read-onlyEndpoints without a control plane are respected as fixed: you pick from what they serve, never push weights at them.
Live checks
  • Status dotsGreen = answering right now, probed live - never assumed. Machines and models re-check when you open Settings.
  • Play protectionPlay refuses to start against a model that is not actually served, and opens the Go step to fix it - no silent zero-scores.
  • Enroll watchAfter generating a bootloader, the machine list watches for the box to call home and it appears by itself.
  • DownloadsBackground weight prefetches tick live on the model row - percent, GB, speed - and the row flips to tap-to-seat when done.
Sharing & upkeep
  • Move setupExport everything - machines, lineup, tasks, preferences - as one file; import it on any other Bake-off. API keys stay behind.
  • UpdatesA banner appears when a new version ships, with what changed. Update now keeps your settings and reloads when done. Settings > General checks on demand.
  • Login & httpsIn run.config.sh set DASH_AUTH="user:pass" and/or DASH_TLS="self", then re-run install.sh.
  • UninstallSettings > General removes the services and launcher; your folder stays for a clean delete.
Tips
  • Steadier scoresSet Runs per question to 3+ when loading a task - more passes smooth accuracy on small sets.
  • Doc baselinesLoading a document task offers a Text baseline picker - what the page readers are compared against: OCR text, parser text, or both.
  • Save the reportThe dock's download button saves a self-contained HTML report - grab it before loading a new task; the live board resets.
  • Phone friendlyThe whole app works from a phone on your network - great as a wall display while a Set runs.
Frequently asked
  • A model is folded away as "set aside"?The loaded set compares the other kind. Tap the folded model to switch the set, or load a matching task.
  • My thinking model scored 0%?Its reasoning likely ate the answer budget. Leave thinking off for comparisons, or raise Answer length when you turn it on.
  • Checked for updates right after a release and it says up to date?Results are cached about an hour server-side and a few minutes on GitHub's side. The Settings button asks fresh; give a new release a few minutes.
  • Where did my results go?Loading a new task or updating clears the live board. Download the report first - it is self-contained.
  • A private GitHub repo will not load?Public links only right now - use the Raw view of a public file, or paste the text itself.
  • I added a docset folder and it is not listed?Document sets are read at startup - restart the dashboard service once.
  • Windows?A jump host usually only needs a browser - run Bake-off on any Linux box and open its address. Native install: the PowerShell line in the README.
  • Does anything leave my network?Your questions and pages never do. The app only calls out when you ask it to - a link you paste, a Hugging Face add or refresh - plus one version check a day.
  • Why is a trending model not offered on my machine?One-tap offers require a vLLM-bootable architecture, no license gate, and enough memory on that box. Unknown or oversized models stay listed without the offer.
  • Are the cost figures real?Local runs cost electricity, not per-token fees. The "frontier cost avoided" line is illustrative at a stated rate - swap in your contracted rate when quoting it.

Adding a machine

A machine is anywhere models answer from: a GPU box on your network running an OpenAI-compatible server (vLLM is the reference setup), or a hosted /v1 service like an NVIDIA NIM. Register it once here - after that, every model pick happens by tapping names, and this URL is never typed again.

Fastest way for your own GPU box: enroll it over ssh. Type its ssh address - joe@192.168.128.200, admin@gpu-node-7, or an alias from your ssh config - and press Enroll. If this computer does not have key access yet, you will be shown the one command to run (ssh-copy-id joe@192.168.128.200, password typed once) and a Recheck button.

What happens to the machine: the dashboard connects over ssh and does everything a careful engineer would - it looks at the GPUs, installs the vLLM serving layer in its own folder (reusing one if the box already has it), starts a small control agent per GPU so models can be swapped from here, checks whether the box can reach this dashboard and quietly builds a private tunnel when it cannot, then registers each GPU as a machine below. Every step reports live, and anything that fails says exactly why. Nothing touches the rest of the box, no admin rights are used, and re-enrolling the same box updates it instead of duplicating it.

Done with a box? Its menu has Unenroll & clean box: serving stops, Bake-off's files are removed, the tunnel comes down, and the machine leaves the list - only the downloaded model weights remain, so a future enroll skips the big downloads.

Already set up in another lab? Configure > General > Import / export carries the whole configuration over (minus API keys and ssh details).
The three fields
  1. 1
    Machine nameA friendly label like "gpu-box-1" or "NVIDIA hosted". It appears wherever this machine's models show up, so name it the way your team talks.
  2. 2
    Address (URL)Where the server answers the /v1 API. On your network that looks like http://gpu-box-1:8000/v1; a hosted service looks like https://integrate.api.nvidia.com/v1. mwboot prints the exact address for its box.
  3. 3
    API keyOnly for endpoints that require one - hosted NIMs use keys starting with nvapi-. It is stored with this machine and sent with every request. Machines on your own network usually need none.

Check, then Save. Check asks the address what models it serves and turns green when it answers. Save adds the machine to the registry - its models appear on the shelf, and tapping the machine puts its best model into the selected slot.

What's supported
  • Server softwareAny OpenAI-compatible /v1 server - vLLM, TensorRT-LLM, SGLang, NVIDIA NIM, llama.cpp's llama-server, LM Studio, TGI, or a gateway. If it speaks the OpenAI /v1 API, it works. (LangChain and the like are client libraries, not servers - point at the /v1 endpoint they call.)
  • Vision modelsModels that read images as well as text (Gemma 3, Qwen2.5-VL, Llama 3.2 Vision, Phi-3.5 Vision, and similar) get a Vision badge and can take the Document ingestion tasks - invoices, tables, charts, and forms sent as page images. Serve them the same way; bake-off detects the capability for you.
  • One GPUAn NVIDIA GPU with a current driver. More memory runs a bigger, smarter model; mwboot picks one that fits.
  • Many GPUs or a clusterbake-off points at one endpoint, not at GPUs. The serving framework (vLLM, TensorRT-LLM, SGLang) spreads the model across the cards with tensor / pipeline parallelism and exposes a single /v1 URL - whether that is one GPU, an 8-way HGX tray, or a 72-GPU GB200 NVL72 rack. Start the server with the parallelism you want, give bake-off that URL, and record the setup under Where it runs.
  • RecommendedStart with vLLM and one model per GPU, launched by mwboot so the box gets a swap agent too. That is the setup the model catalog drives.
  • No GPU yet?Point at a hosted OpenAI-compatible service instead: its address goes in the same field, plus the API key it gave you. Its models join the shelf like any other machine's.
